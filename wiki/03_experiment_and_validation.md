# Experiment and Validation

## Purpose of This Document

This document describes how to validate the pipeline before pointing it at a real Salesforce org. It explains what the experiment tests, what datasets are used, how the validation is structured, and what "good enough" looks like before the tool is considered ready for production use.

---

## What the Experiment Tests

The pipeline has two components that require validation before they can be trusted in production:

**The Classifier** — ten rule-based checks written in Python. These are deterministic and testable directly. The experiment verifies that every check fires on the cases it is designed to catch and does not fire on cases it should leave alone. The rules are described in `02_how_it_works.md`.

**The Prompts** — two prompt templates (`prompt_a_flagged_fields.md` and `prompt_b_uncertain_fields.md`) that instruct the LLM how to handle each classification bucket. These are not deterministic. The experiment surfaces what good and bad output looks like, and the prompts are revised until output quality is consistently acceptable.

The experiment is not about achieving a perfect score. It is about building enough evidence that the Classifier catches the right things and the prompts produce suggestions a human reviewer can actually use.

---

## What the Experiment Does NOT Test

- Real Salesforce connectivity (Tooling API or Metadata API)
- Enterprise LLM infrastructure (Azure OpenAI, AWS Bedrock, Google Vertex AI)
- Write-back to a live org

These are intentionally excluded. The experiment uses synthetic data and a local or simulated LLM so that the pipeline can be run and re-run publicly, without credentials, without cost, and without risk to any org.

---

## Experiment Mode vs Production Mode

The experiment runs the **same scripts** as production. There is no separate experiment codebase. Two seams in the pipeline are swapped via `config.yml`:

| Seam | Experiment Mode | Production Mode |
|---|---|---|
| Data input | Reads from a local JSON file (`sf_metadata_raw_training.json` or `sf_metadata_raw_test.json`) | Calls Salesforce Tooling API |
| LLM calls | Calls AOAI API Simulator (local) or Ollama (local) | Calls enterprise LLM (Azure OpenAI, Bedrock, Vertex AI) |
| Write-back (Script 2) | Dry-run — logs decisions but does not call Metadata API | Calls Salesforce Metadata API |

Everything between those seams — classification logic, routing, prompt assembly, batching, response parsing, review file generation, decision validation — runs identically in both modes.

This means the experiment validates the real pipeline, not a toy version of it.

## LLM Provider Selection in `config.yml`

```yaml
mode: experiment        # experiment | production

data_source:
  experiment_file: data/sf_metadata_raw_training.json   # switch to sf_metadata_raw_test.json for final validation

llm:
  provider: aoai_simulator    # options: aoai_simulator | ollama | azure_openai | openai
  endpoint: http://localhost:8080
  model: gpt-4o

objects:
  - Account
  - Contact
  - Opportunity
  - Case
```

Setting `mode: experiment` tells Script 2 to run in dry-run — decisions are validated and logged, but no call is made to the Salesforce Metadata API.

## Synthetic Datasets — Training and Test

The experiment uses two synthetic datasets. Both are committed to the repository and safe to share publicly. Neither contains real org data.

### Why Two Datasets

The validation approach mirrors standard machine learning practice: a **training set** is used to develop and tune the system, and a held-out **test set** is used only once — to verify that the tuned system generalises to data it has never seen. This separation is essential. If the same dataset is used for both tuning and validation, there is no way to know whether the pipeline works in general or has simply been over-fitted to the examples used during development.

### Training Set — `data/sf_metadata_raw_training.json`

**288 fields — 4 objects × 72 fields each**

Objects: Account, Contact, Opportunity, Case.

The training set is the primary working dataset. It is used during all iterations of classifier development and prompt refinement. Every classification rule is deliberately exercised across multiple field examples, with sufficient variety in phrasing to confirm that the classifier responds to the underlying pattern, not just a single form of wording.

Field composition per object (72 fields):

| Status | Count | Purpose |
|---|---|---|
| FLAGGED | 16 | 2–3 distinct examples per rule (R1–R5), including multi-rule edge cases where the first-firing rule determines the status |
| UNCERTAIN | 13 | 2–3 distinct examples per rule (R6–R10), including multi-rule edge cases |
| PASSED | 39 | Clean fields (genuine good descriptions) and near-miss fields (descriptions that look like they might trigger a rule but correctly pass) |
| SKIPPED | 4 | System fields not writable via the Metadata API |
| **Total** | **72** | |

Near-miss fields are a critical part of the training set. They are PASSED fields deliberately designed to look similar to a failing case — for example a description that contains a known acronym but expands it inline, or a picklist description that is short but still enumerates all values. They test that the classifier does not over-fire.

### Test Set — `data/sf_metadata_raw_test.json`

**144 fields — 4 objects × 36 fields each**

Objects: Account, Contact, Opportunity, Case.

The test set is held out until the classifier and prompts are considered stable on the training set. It is not used during development. It is run exactly once as a final validation check.

Field composition per object (36 fields):

| Status | Count | Purpose |
|---|---|---|
| FLAGGED | 8 | At least one example per rule (R1–R5) on unseen field names and descriptions |
| UNCERTAIN | 6 | At least one example per rule (R6–R10) on unseen field names and descriptions |
| PASSED | 20 | Mix of clean and near-miss fields, distinct from training set examples |
| SKIPPED | 2 | System fields |
| **Total** | **36** | |

The test set uses the same objects and the same rule distribution as the training set, but all field names, descriptions, and error phrasings are different. If the classifier and prompts perform well on the test set, the experiment is considered complete.

### Rule Coverage Across Both Datasets

The table below reflects the updated rule numbering. Rules 1–5 produce FLAGGED (Step 1). Rules 6–10 produce UNCERTAIN (Step 2).

| Rule | Name | Step | Status |
|---|---|---|---|
| R1 | NULL / Blank | Step 1 | FLAGGED |
| R2 | Echo / Too Short | Step 1 | FLAGGED |
| R3 | Wrong Type Hint | Step 1 | FLAGGED |
| R4 | Undefined Picklist | Step 1 | FLAGGED |
| R5 | Stale or Placeholder | Step 1 | FLAGGED |
| R6 | Too Long (>200 chars) | Step 2 | UNCERTAIN |
| R7 | Jargon Without Context | Step 2 | UNCERTAIN |
| R8 | Contradictory | Step 2 | UNCERTAIN |
| R9 | Audience Mismatch | Step 2 | UNCERTAIN |
| R10 | Ambiguous | Step 2 | UNCERTAIN |

## How to Run the Experiment

### Prerequisites

- Python 3.x installed
- Dependencies installed: `pip install -r requirements.txt`
- One of the following LLM options running locally:
  - AOAI API Simulator: follow setup instructions at https://github.com/microsoft/aoai-api-simulator
  - Ollama: install from https://ollama.com and pull a model (e.g. `ollama pull llama3`)
- `config.yml` created by copying and editing `config.example.yml`

### Phase 1 — Training (iterate until stable)

These steps are repeated as many times as needed until the classifier and prompts meet the Definition of Done criteria on the training set.

1. Set `data_source.experiment_file: data/sf_metadata_raw_training.json` in `config.yml`
2. Run Script 1: `python scripts/01_ingest_classify_send.py`
3. Check `data/sf_classified.json` — verify the classifier is firing correctly (see Classifier Validation below)
4. Open `data/review_queue_{timestamp}.xlsx` — review Tab A and Tab B, approve / edit / reject each row
5. Save the file
6. Run Script 2: `python scripts/02_deploy_approved.py`
7. Review `data/write_log_{timestamp}.xlsx` — no changes will have been written to Salesforce
8. If classifier accuracy or LLM output quality is below the Definition of Done thresholds, revise the relevant classifier logic or prompt file and repeat from Step 2

### Phase 2 — Test (run once, do not iterate)

These steps are run exactly once, after Phase 1 is complete and the classifier and prompts are considered stable.

1. Set `data_source.experiment_file: data/sf_metadata_raw_test.json` in `config.yml`
2. Run Script 1: `python scripts/01_ingest_classify_send.py`
3. Check `data/sf_classified.json` — verify classifier accuracy on unseen fields
4. Open `data/review_queue_{timestamp}.xlsx` — review Tab A and Tab B
5. Save the file
6. Run Script 2: `python scripts/02_deploy_approved.py`
7. Review `data/write_log_{timestamp}.xlsx`
8. Record results in the Test Results section at the end of this document

**The test set must not be used to tune the classifier or prompts.** If results are poor, the correct response is to return to Phase 1, identify the gap, iterate on the training set, and run Phase 2 again from scratch.

## Classifier Validation

After each Script 1 run, verify the classifier is firing correctly by inspecting `data/sf_classified.json`. For each rule, confirm:

- Fields designed to trigger that rule are assigned the expected status (FLAGGED or UNCERTAIN)
- Near-miss fields designed to pass are assigned PASSED
- System fields are assigned SKIPPED
- Rules 1–5 never produce UNCERTAIN
- Rules 6–10 never produce FLAGGED
- In multi-rule cases, the first-firing rule determines the status — FLAGGED takes priority over UNCERTAIN because Step 1 is evaluated before Step 2

If a rule misfires — either missing a case it should catch or firing on a case it should not — the classifier logic in `01_ingest_classify_send.py` should be corrected before evaluating LLM output.

## Prompt Evaluation

Once the classifier is confirmed correct on the training set, evaluate the LLM suggestions in the review queue against the criteria below. If output quality is consistently poor for a given rule or category, revise the relevant prompt file and re-run Phase 1.

### What "Good Output" Means

A suggested description is considered acceptable if it meets all of the following:

**Length**
- Minimum 30 characters
- Maximum 255 characters (hard Salesforce limit)
- Target range for a well-written description: 60–180 characters

**Clarity — no vague terms**

Descriptions are rejected if they contain terms that add no information:
- "used for", "used by", "used to"
- "relevant", "applicable", "various", "related to"
- "information", "data", "details" used without qualification
- "TBD", "N/A", "see above", "ask", "deprecated"

**Specificity**

The description must remain meaningful if the field name and object name are removed. A description that could apply to any field in any system has failed the specificity test.

**Audience**

The description must be written for systems, integrations, and developers — not for end users. It must not contain UI instructions, help text language, or navigational cues.

**Picklist coverage**

If the field is a Picklist or Multi-Select Picklist, the description must reference the values or explain what the values represent.

**Type alignment**

The description must be consistent with the field type. A Checkbox must not be described as if it stores free text. A Date field must not be described as storing a range.

**For UNCERTAIN / R6 (Too Long) specifically**

The LLM suggestion must be shorter than the original and within the 30–200 character target range. The core meaning must be preserved — the LLM must not drop specificity in order to shorten. If the original description is accurate and specific but long, a shortened version that loses key information should be rejected.

## Definition of Done

The experiment is complete when all of the following are true.

### Phase 1 — Training Set

**Classifier**
- All 10 rules fire correctly on the training set
- No field is misclassified on a case explicitly designed to test that rule
- No Step 1 rule (R1–R5) produces UNCERTAIN
- No Step 2 rule (R6–R10) produces FLAGGED
- In all multi-rule cases, the first-firing rule is recorded and the status follows that rule's step

**LLM Output — Tab A (FLAGGED fields)**
- At least 80% of suggestions are approvable without editing
- No suggestion contains vague terms from the rejection list above
- No suggestion exceeds 255 characters
- No suggestion contradicts the field type

**LLM Output — Tab B (UNCERTAIN fields)**
- LLM correctly identifies genuinely acceptable descriptions and does not suggest unnecessary rewrites
- For R6 (Too Long): LLM shortens the description without losing specificity or accuracy
- Where a rewrite is suggested for any other rule, it meets the same quality bar as Tab A

**Human Review**
- Every row in Tab A and Tab B has a recorded decision (Approve / Edit / Reject)
- No row is left blank
- At least one full review cycle has been completed end-to-end: Script 1 → review file → Script 2 → write log

### Phase 2 — Test Set

**Classifier**
- All 10 rules fire correctly on unseen fields
- No misclassification on any field in the test set

**LLM Output**
- At least 80% of Tab A suggestions are approvable without editing
- At least 80% of Tab B suggestions are approvable without editing (or correctly preserved without change)

**Reproducibility**
- A second person can clone the repository, configure `config.yml`, and run the full experiment — both phases — without any guidance beyond this wiki and the README
