## Experiment and Validation

### Purpose of This Document

This document describes how to validate the pipeline before pointing it at a real Salesforce org. It explains what the experiment tests, what tech stack it uses, how to run it, and what "good enough" looks like before the tool is considered ready for production use.

\---

### What the Experiment Tests

The pipeline has two components that require validation before they can be trusted in production:

**The Classifier** — ten rule-based checks written in Python. These are deterministic and testable directly. The experiment verifies that every check fires on the cases it is designed to catch and does not fire on cases it should leave alone. The rules are described in `02_how_it_works.md`

**The Prompts** — two prompt templates (`prompt_a_flagged_fields.md` and `prompt_b_uncertain_fields.md`) that instruct the LLM how to handle each classification bucket. These are not deterministic. The experiment surfaces what good and bad output looks like, and the prompts are revised until output quality is consistently acceptable.

The experiment is not about achieving a perfect score. It is about building enough evidence that the Classifier catches the right things and the prompts produce suggestions a human reviewer can actually use.

\---

### What the Experiment Does NOT Test

* Real Salesforce connectivity (Tooling API or Metadata API)
* Enterprise LLM infrastructure (Azure OpenAI, AWS Bedrock, Google Vertex AI)
* Write-back to a live org

These are intentionally excluded. The experiment uses synthetic data and a local or simulated LLM so that the pipeline can be run and re-run publicly, without credentials, without cost, and without risk to any org.

\---

## Experiment Mode vs Production Mode

The experiment runs the **same scripts** as production. There is no separate experiment codebase. Two seams in the pipeline are swapped via `config.yml`:

|Seam|Experiment Mode|Production Mode|
|-|-|-|
|Data input|Reads from `data/sf_metadata_raw.json` (pre-generated synthetic file)|Calls Salesforce Tooling API|
|LLM calls|Calls AOAI API Simulator (local) or Ollama (local)|Calls enterprise LLM (Azure OpenAI, Bedrock, Vertex AI)|
|Write-back (Script 2)|Dry-run — logs decisions but does not call Metadata API|Calls Salesforce Metadata API|

Everything between those seams — classification logic, routing, prompt assembly, batching, response parsing, review file generation, decision validation — runs identically in both modes.

This means the experiment validates the real pipeline, not a toy version of it.

## Experiment Tech Stack

|Component|Tool|Notes|
|-|-|-|
|Language|Python 3.x|Same as production|
|Input data|`data/sf_metadata_raw.json`|Synthetic — safe to share publicly|
|LLM option A|[AOAI API Simulator](https://github.com/microsoft/aoai-api-simulator)|Simulates Azure OpenAI API locally. No API key required.|
|LLM option B|[Ollama](https://ollama.com/)|Runs open-source models locally. No API key required. Fully offline.|
|Review file|`review_queue_{timestamp}.xlsx`|Generated by Script 1, same format as production|
|Write log|`write_log_{timestamp}.xlsx`|Generated by Script 2 in dry-run mode|
|Config|`config.yml`|Copied from `config.example.yml` — not committed to version control|

## LLM Provider Selection in `config.yml`

```yaml
mode: experiment        # experiment | production

data\_source:
  experiment\_file: data/sf\_metadata\_raw.json

llm:
  provider: aoai\_simulator    # options: aoai\_simulator | ollama | azure\_openai | openai
  endpoint: http://localhost:8080
  model: gpt-4o

objects:
  - Account
  - Contact
  - Case
```

Setting `mode: experiment` tells Script 2 to run in dry-run — decisions are validated and logged, but no call is made to the Salesforce Metadata API.

## Synthetic Input Dataset

The input file (`data/sf_metadata_raw.json`) is a generated dataset **designed to exercise every classification rule.** It contains a deliberate mix of:

* Fields with no description (Rule 1 — NULL)
* Fields whose description echoes the field name (Rule 2 — Echo / Too Short)
* Fields whose description contradicts the field type (Rule 3 — Wrong Type Hint)
* Picklist fields with no reference to their values (Rule 4 — Undefined Picklist)
* Descriptions that are too long (Rule 5 — Too Long)
* Placeholder or stale descriptions (Rule 6 — Stale or Placeholder)
* Descriptions containing unexplained internal acronyms (Rule 7 — Jargon Without Context)
* Descriptions that contradict observable field configuration (Rule 8 — Contradictory)
* Descriptions written for end users rather than systems (Rule 9 — Audience Mismatch)
* Descriptions that are technically present but meaningless without context (Rule 10 — Ambiguous)
* Fields with genuinely good descriptions (expected to **PASS**)
* System fields that cannot be written via the Metadata API (expected to be **SKIPPED**)

This dataset is committed to the repository and safe to share publicly. It contains no real org data.

## How to Run the Experiment

### Prerequisites

* Python 3.x installed
* Dependencies installed: `pip install -r requirements.txt`
* One of the following LLM options running locally:

  * AOAI API Simulator: follow setup instructions at https://github.com/microsoft/aoai-api-simulator
  * Ollama: install from https://ollama.com and pull a model (e.g. `ollama pull llama3`)
* `config.yml` created by copying and editing `config.example.yml`

### Steps

1. Copy `config.example.yml` to `config.yml` and set `mode: experiment` and your chosen LLM provider
2. Run Script 1: `python scripts/01_ingest_classify_send.py`
3. Open the generated `data/review_queue_{timestamp}.xlsx`
4. Review Tab A and Tab B — approve, edit, or reject each row
5. Save the file
6. Run Script 2: `python scripts/02_deploy_approved.py`
7. Review the generated `data/write_log_{timestamp}.xlsx` — no changes will have been written to Salesforce

\---

## Classifier Validation

Before reviewing LLM output, verify the Classifier is firing correctly by checking `data/sf_classified.json` after Step 2. For each rule, confirm:

* Fields designed to trigger that rule are assigned the expected status (FLAGGED or UNCERTAIN)
* Fields designed to pass are assigned PASSED
* System fields are assigned SKIPPED
* No field is assigned a status lower than expected when multiple rules apply (FLAGGED takes priority over UNCERTAIN)

If a rule misfires — either missing a case it should catch or flagging a case it should not — the classifier logic in `01_ingest_classify_send.py` should be corrected before evaluating LLM output.

\---

## Prompt Evaluation

Once the Classifier is confirmed correct, evaluate the LLM suggestions in the review queue against the criteria below. If output quality is consistently poor for a given category, revise the relevant prompt file and re-run.

### What "Good Output" Means

A suggested description is considered acceptable if it meets all of the following:

**Length**

* Minimum 30 characters
* Maximum 255 characters (hard Salesforce limit)
* Typical good descriptions fall between 60–180 characters

**Clarity — no vague terms**
Descriptions are rejected if they contain terms that add no information:

* "used for", "used by", "used to"
* "relevant", "applicable", "various", "related to"
* "information", "data", "details" used without qualification
* "TBD", "N/A", "see above", "ask", "deprecated"

**Specificity**
The description must remain meaningful if the field name and object name are removed. A description that could apply to any field in any system has failed the specificity test.

**Audience**
The description must be written for systems, integrations, and developers — not for end users. It must not contain UI instructions, help text language, or navigational cues.

**Picklist coverage**
If the field is a Picklist, the description must reference the values or explain what the values represent.

**Type alignment**
The description must be consistent with the field type. A Checkbox must not be described as if it stores freeform text.

## Definition of Done

The experiment is complete when all of the following are true:

**Classifier**

* All 10 rules fire correctly on the synthetic dataset
* No field is misclassified on a case explicitly designed to test that rule
* FLAGGED takes priority over UNCERTAIN in all multi-rule cases

**LLM Output — Tab A (FLAGGED fields)**

* At least 80% of suggestions are approvable without editing
* No suggestion contains vague terms from the rejection list
* No suggestion exceeds 255 characters
* No suggestion contradicts the field type

**LLM Output — Tab B (UNCERTAIN fields)**

* LLM correctly identifies genuinely acceptable descriptions and does not suggest unnecessary rewrites
* Where a rewrite is suggested, it meets the same quality bar as Tab A

**Human Review**

* Every row in Tab A and Tab B has a recorded decision (Approve / Edit / Reject)
* No row is left blank
* At least one full review cycle has been completed end-to-end: Script 1 → review file → Script 2 → write log

**Reproducibility**

* A second person can clone the repository, configure `config.yml`, and run the full experiment without any guidance beyond this wiki and the README
